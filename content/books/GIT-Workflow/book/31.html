3.1 Working with Release Branches

Branch from: stage

Merge into: stage and master

Naming convention: release-*

3.1.1 Workflow:

Create a release branch. Pay attention to the naming convention.

git checkout -b release-x.y.z stage

This creates the branch and switches you into it.

x - major version

y - minor version

z - hotfix (bug fixes)

2) Do some coding here...

Append a note about this release version to the readme file for the project.

echo "released version x.y.z" >> readme.md

This assumes you have a readme.md file for the project (which you should) and appending this statement to the file makes sense. So maybe your readme.md file reads like this:

# Big Project
 
## How it works
 
blah blah blah
 
## Credits
 
people, people, people
 
...  lots of other stuff ...
 
## Release History
 
released version 1.0.0
released version 1.0.1
released version 1.1.0

Add and commit changes to the feature branch as you go along

git add .

and

git commit

Now, you might leave this open for a few days to give developers time to apply some last minute hot fixes to bugs or add magical qualities to the code...

Those lasst minute tweaks should be added and commited.

Merge the release branch with the master branch (go live!)

switch to master branch

git checkout master

merge release with master

git merge --no-ff release-x.y.z

Tell git what the current release is

git tag -a x.y.z

Merge the release branch with the stage branch (keep that development team up to date!)

switch to stage branch

git checkout stage

merge release with stage

git merge --no-ff release-x.y.z

Lastly, delete the release branch

git branch -d release-x.y.z
